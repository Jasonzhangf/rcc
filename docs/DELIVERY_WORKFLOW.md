# Delivery Workflow

## 索引概要
- L1-L7 `purpose`：本文件定义单个任务的交付步骤。
- L9-L19 `sequence`：任务从开始到关闭的顺序。
- L21-L33 `artifacts`：每一步必须留下的产物。
- L35-L44 `reporting`：交付汇报格式。
- L46-L49 `close-rule`：关闭规则。

## 目标
把每个任务的交付过程固定下来，避免出现“文档未定、skills 未沉淀、测试未跑就宣布完成”的情况。

## 标准交付顺序
1. 在任务系统中建立或认领任务。
2. 锁定 docs 真源，补全目标/边界/完成标准。
3. 创建或更新关联 skill。
4. 做当前阶段的最小实现。
5. 运行验证脚本或测试。
6. 用证据更新任务状态。
7. 关闭任务并指向下一阶段入口。

## 每一步的必备产物
### 任务系统
- issue / epic / 子任务
- 当前状态
- 完成标准

### Docs
- 更新的真源文档
- 入口路由更新（若涉及）

### Skills
- 更新的 `SKILL.md`
- 与 docs 的边界说明

### Verification
- 命令
- 输出摘要
- 失败时暴露的缺口

## 交付汇报格式
1. **变更**：新增/修改了哪些文件。
2. **验证**：运行了哪些命令，结果如何。
3. **结论**：当前阶段是否闭环完成。
4. **下一步**：下一阶段入口文档或任务。

## 关闭规则
- 只有在 docs、skills、验证证据都齐全时，当前任务才能关闭。
- 若验证未通过，任务保持进行中，不以口头解释代替关闭。
